# Tailscale OpenClash Bypass Design

## Status

Design approved on 2026-07-16; final-review corrections approved on 2026-07-17.

## Background

OpenClash can intercept traffic generated by `tailscaled` before Tailscale completes
peer-to-peer endpoint negotiation. The affected traffic falls into two groups:

- locally generated Tailscale outer UDP and TCP traffic, identified by Tailscale's
  `0x80000` packet mark;
- traffic entering the router from `tailscale0` before it is forwarded to a local
  network or subnet route.

The office router currently bypasses OpenClash with four manually maintained
nftables rules in OpenClash's developer custom firewall script. The rules work,
but the installation is device-specific and is not owned by the customized
`luci-app-tailscale` package.

The package already has a separate WAN-direct feature that manages firewall UCI
configuration. That feature and the OpenClash bypass must remain independent.
The OpenClash bypass must not use a firewall4 reload as a lifecycle mechanism.

## Goals

- Automatically install and maintain the four required bypass rules when the
  customized package is installed on an OpenWrt device using OpenClash.
- Use OpenClash's supported developer custom firewall script hook so rules are
  recreated after OpenClash creates its built-in nftables chains.
- Preserve all user-owned content in the OpenClash custom firewall script.
- Make applying and removing rules idempotent and ownership-safe.
- Provide a LuCI option that is enabled by default when OpenClash is present and
  can be explicitly disabled.
- Support only firewall4/nftables.
- Keep the WAN-direct and OpenClash-bypass lifecycles completely independent.

## Non-Goals

- Supporting firewall3 or legacy iptables.
- Fixing OpenClash's own firewall-only reload behavior.
- Monitoring or repairing arbitrary firewall4 reloads initiated outside this
  feature.
- Restarting, enabling, disabling, or otherwise managing the OpenClash service.
- Replacing OpenClash's native local-network destination bypass logic.
- Dynamically synchronizing Tailscale routing table 52 into OpenClash rules.
- Modifying `/etc/config/firewall` for the OpenClash bypass.

## Hard Boundary Between Features

The package contains two separate features with no calls between them.

### WAN-Direct Flow

The existing WAN-direct feature owns only its firewall UCI objects and its
firewall4 reload behavior.

```text
WAN-direct setting changes
  -> update firewall UCI objects
  -> reload firewall4
  -> verify WAN-direct firewall state
```

The WAN-direct flow must not:

- edit the OpenClash custom firewall script;
- invoke the OpenClash bypass helper;
- inspect OpenClash nftables chains;
- restart or reload OpenClash;
- repair OpenClash after a firewall4 reload.

### OpenClash-Bypass Flow

The new OpenClash-bypass feature owns only its hook block and four runtime
nftables rules.

```text
OpenClash creates its built-in nftables chains
  -> OpenClash executes its developer custom firewall script
  -> the managed hook invokes the package helper
  -> the helper reconciles four bypass rules
```

The OpenClash-bypass flow must not:

- read or write firewall UCI objects;
- edit `/etc/config/firewall`;
- reload or restart firewall4;
- invoke the WAN-direct helper or apply path;
- restart, reload, enable, or disable OpenClash.

Changing settings in one flow must not execute any action from the other flow.
If both settings are changed in one LuCI submission, each flow executes only its
own independent apply operation.

## Components

### UCI Setting and LuCI Control

Add an OpenClash-bypass option to the Tailscale application's own UCI section.
Its effective default is enabled when OpenClash is installed. When OpenClash is
not installed, the feature silently performs no action and does not create an
OpenClash directory or custom script.

The LuCI control supports these operational states:

- enabled and active;
- enabled and waiting for OpenClash chains;
- disabled;
- unsupported because the device is not using firewall4/nftables;
- configuration error requiring user attention.

The setting is independent from all WAN-direct options.

### Package-Owned Helper

Install an executable helper at:

```text
/usr/sbin/tailscale_openclash_bypass
```

The helper exposes these operations:

- `sync`: re-read the app-owned UCI setting and reconcile the complete enabled or
  disabled state under one exclusive lock;
- `reconcile-hook`: install or update the managed hook block without changing
  user content; this remains a focused diagnostic/test operation and is not used
  by the procd lifecycle;
- `apply`: reconcile the four runtime nftables rules;
- `cleanup`: remove only the managed hook block and managed nftables rules;
- `status`: return a machine-readable operational state for LuCI and tests.

All mutating operations are idempotent and use an OpenWrt-compatible lock. The
procd service invokes only `sync`, so one UCI decision, hook reconciliation, and
runtime-rule reconciliation cannot be interleaved with a concurrent enable,
disable, OpenClash startup, or LuCI apply.

### Managed OpenClash Hook

The helper manages one uniquely delimited block in:

```text
/etc/openclash/custom/openclash_custom_firewall_rules.sh
```

The block contains only a guarded invocation of the package helper:

```sh
# BEGIN luci-app-tailscale 托管：Tailscale 绕过 OpenClash
if test -x /usr/sbin/tailscale_openclash_bypass; then
    /usr/sbin/tailscale_openclash_bypass apply
fi
# END luci-app-tailscale 托管：Tailscale 绕过 OpenClash
```

Hook reconciliation must:

- preserve all bytes outside the managed block;
- preserve the file's mode and ownership;
- identify markers only when an entire line exactly equals the marker, using
  BusyBox 1.36-compatible tools; `grep -b` is not available on the target;
- validate the complete managed body and replace a stale managed body with the
  canonical block;
- write through a temporary file and atomically replace the original only after
  `sh -n` succeeds;
- restore the original immediately if validation or replacement fails;
- refuse to modify a file containing malformed or duplicate managed markers;
- create no file when OpenClash is absent;
- create a minimal executable custom script only when OpenClash is installed but
  the documented hook path is missing.

Temporary hook and nftables files are tracked explicitly and removed by signal
and exit traps. Replacement restores numeric ownership before restoring mode so
`chown` cannot clear special mode bits after `chmod`.

The marker comments and nftables rule comments are written in Chinese, with a
stable `luci-app-tailscale` ownership prefix for reliable cleanup.

## Managed nftables Rules

The helper manages exactly these four rules in table `inet fw4`:

| Chain | Match | Action |
| --- | --- | --- |
| `openclash_mangle_output` | `meta mark & 0x00ff0000 == 0x00080000` | `return` |
| `openclash_output` | `meta mark & 0x00ff0000 == 0x00080000` | `return` |
| `openclash_mangle` | `iifname "tailscale0"` | `return` |
| `openclash` | `iifname "tailscale0"` | `return` |

Each rule includes a counter and a unique comment beginning with
`luci-app-tailscale:`. The rules are inserted at the head of their respective
OpenClash chains so later interception rules cannot run first.

Before applying, the helper verifies that firewall4/nftables and all four target
chains exist. It then reconciles the complete rule set as one operation. It does
not leave a partial rule set when any target chain is missing.

Rules are identified for update and removal only by their unique ownership
comments. The helper does not remove equivalent rules written by the user or
another package. Repeated application leaves exactly one owned rule in each
target chain.

An absent `inet fw4` table or an already removed target chain is a successful
cleanup no-op. If a chain disappears between a table snapshot and a chain read,
the helper refreshes table state before classifying the result as waiting,
unsupported, or an actual error.

## Existing Site-to-Site SNAT Contract

The existing `子网互通` option remains outside the OpenClash flow. It controls
both Tailscale `--snat-subnet-routes` and the app-owned firewall4 zone's `masq`
option. Enabling it means `--snat-subnet-routes=false` and
`firewall.tszone.masq='0'`.

The `tailscale` firewall zone exists whenever either forwarding access rules are
selected or site-to-site no-SNAT is enabled. Forwarding sections are still
created only for selected access directions. Therefore no-SNAT remains effective
when the access list is empty, without creating unrelated forwarding rules.

## Destination Bypass Responsibility

The helper does not create destination-based rules for Tailnet addresses or
advertised subnet routes. OpenClash's native local-network handling remains
responsible for those destinations.

In particular, the feature does not enumerate routes from Tailscale routing table
52 and does not add `100.64.0.0/10` or private subnet routes to OpenClash. This
keeps route ownership with Tailscale and destination bypass ownership with
OpenClash.

## Lifecycle

### Package Install or Upgrade

1. Use the standard LuCI package post-install path so cache invalidation and
   `rpcd` reload remain intact; do not add a package-specific start/enable hook.
2. OpenWrt's `default_postinst` starts/enables the packaged init script.
3. The one-shot init script invokes `sync`, which re-reads UCI under one lock.
4. If OpenClash is absent, finish without creating a hook or changing rules.
5. If firewall4/nftables is unsupported, report `unsupported` and make no hook or
   rule change.
6. When enabled and firewall4 exists, reconcile the managed hook. If all four
   OpenClash chains exist, reconcile the rules in one nft transaction.
7. If the chains do not exist, retain the canonical hook and report `waiting`;
   do not start or restart OpenClash. Its next successful startup executes the
   hook.

### OpenClash Start or Restart

OpenClash creates its built-in rules and then executes its developer custom
firewall script. The managed block calls `apply`, which validates the target
chains and reconciles the four rules. No firewall4 reload occurs.

### Enable Through LuCI

Enabling follows the same steps as package installation: reconcile the hook and
apply immediately only when all target chains already exist. Missing chains
produce the `waiting` state without managing the OpenClash service.

### Disable Through LuCI

Disabling removes the managed hook block and the four owned runtime rules. It
does not remove user rules, reload firewall4, or restart OpenClash.

### Package Removal

The package removal hook invokes `cleanup` while the helper is still present.
Cleanup is best-effort and ownership-scoped. An absent OpenClash installation,
absent chains, or an already removed hook is a successful no-op.

## Failure Handling

- Missing OpenClash is a silent no-op, not an error.
- Unsupported firewall backend produces an explicit unsupported status and no
  changes.
- Missing target chains produce a waiting status and no partial rules.
- Malformed or duplicate managed hook markers produce an error and leave the
  custom script unchanged.
- nftables command failures leave a nonzero status and are logged with `logger`.
- The helper never responds to a failure by reloading firewall4 or restarting
  OpenClash.
- LuCI displays the helper result without reporting success when hook or rule
  reconciliation failed.
- UCI booleans use OpenWrt semantics: `0`, `false`, `off`, `no`, and `disabled`
  are false; known true aliases are true; malformed values do not silently enable
  the feature.

OpenClash may fail to rebuild its own chains after a firewall4 reload. That is an
independent OpenClash lifecycle problem. This feature reports missing chains and
waits for a successful OpenClash startup; it does not cross into service repair.

## Testing

### Automated Tests

Add shell and LuCI tests covering:

- default-enabled detection when OpenClash is installed;
- silent no-op behavior when OpenClash is absent;
- rejection of non-firewall4 backends;
- hook insertion while preserving user content, mode, and ownership;
- hook update and repeated reconciliation without duplication;
- malformed and duplicate marker rejection without file changes;
- hook cleanup without removing surrounding user content;
- exact generation of all four nftables rules;
- insertion at the head of each target chain;
- unique Chinese ownership comments and counters;
- repeated apply leaving one owned rule per chain;
- no partial application when a chain is missing;
- cleanup removing only owned rules;
- package install, upgrade, disable, and removal lifecycle behavior;
- LuCI status and toggle behavior;
- static assertions that the OpenClash helper contains no firewall UCI writes,
  firewall reloads, OpenClash service operations, or WAN-direct calls;
- static assertions that the WAN-direct apply path contains no OpenClash bypass
  calls or OpenClash service repair.

Existing WAN-direct tests remain separate and must continue to pass unchanged.

### Device Verification

Remote device verification is a separate, explicitly approved operation. It
must verify:

1. Existing user content in the custom script survives migration.
2. Enabling the feature creates one managed hook block.
3. The four rules are present at the heads of the expected chains.
4. Reapplying does not create duplicates.
5. A normal OpenClash restart recreates all four rules through the official hook.
6. OpenClash proxy traffic remains functional.
7. Tailscale outer traffic bypasses OpenClash and cold peer negotiation reaches a
   direct path when the network endpoints permit it.
8. Disabling removes only the package-owned hook and rules.

Device verification for this flow must not reload firewall4. WAN-direct behavior
is verified independently.

## Rollback

The operational rollback is to disable the OpenClash-bypass option, verify that
cleanup removed the owned hook and rules, and then reinstall the previous package
version if a package rollback is required. Cleanup preserves user content.
Because the feature does not write firewall UCI or manage services, rollback
requires no firewall4 or OpenClash restart.

## Documentation After Implementation

After package deployment and device verification:

- replace the office router's manual OpenClash rule procedure with the
  package-managed state;
- remove obsolete manual rollback instructions that apply only to the old
  hand-maintained script;
- document the LuCI toggle, supported backend, managed hook path, status states,
  and ownership-scoped rollback;
- keep the WAN-direct and OpenClash-bypass documentation in separate sections.
